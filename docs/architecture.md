# アーキテクチャ設計書

草野球チーム運営支援システムのアーキテクチャを、アウトサイドインで設計する。

---

## 0. プロダクトビジョン

### 何を売るのか

ツールではない。**草野球チーム運営のテンプレート**をシステムに焼き込んだもの。

既存ツール（調整さん、伝助、Play等）は「箱」を提供するだけで、使い方は代表が自分で考える。
このプロダクトは**運営フロー自体が組み込まれている**。代表は初期設定をするだけで、あとはシステムが回る。

```
初期設定（1回だけ）:
  メンバー登録 / 助っ人候補登録 / グラウンド対象設定 / 精算ルール設定

毎週（代表がやること）:
  「試合を作成」→ 以降は自動

自動で回るもの:
  出欠通知 → 段階的リマインド → 締切処理 → 人数判定 →
  助っ人打診 → 精算計算 → 精算通知

代表が判断するとき:
  Claude Codeに聞く（予測・推薦・文面生成）
```

### なぜ価値があるか

草野球チーム運営の悩みは驚くほど共通している。出欠が返ってこない、人数足りない、
グラウンド確保が大変、精算がめんどくさい、代表が孤軍奮闘。全チーム共通の課題。

このプロダクトは、実際のチーム運営（Xeros: 2003年設立、年間50試合以上）で蓄積された
運営ノウハウをテンプレートとしてシステムに焼き込んでいる。新しいチームの代表が
使い始めた瞬間に、ベストプラクティスが自動的に適用される。

### 通知チャネル戦略

LINE Messaging APIの無料枠は月200通。メンバー15人×月4試合のリマインドでギリギリ。
費用をかけずにスケールするため、メール（Gmail）を併用する。

| 通知種別 | チャネル | 理由 |
|---------|---------|------|
| 初回出欠依頼 | LINE Push | 気づきやすい。月4回×15人=60通 |
| リマインド | メール + Webリンク | LINE通数を消費しない |
| 精算通知 | メール + Webリンク | 金額詳細はメールの方が見やすい |
| 緊急連絡(雨天中止等) | LINE Push | 即時性が必要 |

メールからの回答はWebページ上の1タップで完結する（チャット風UI不要、ボタン1つ）。

### スマホアプリ戦略

LINE通数制限の根本的解決として、メンバー向けスマホアプリを提供する。
プッシュ通知が無料・無制限で使えるため、通知チャネルの制約がなくなる。

| 対象 | インターフェース |
|------|----------------|
| 代表 | Web Dashboard (Next.js) + Claude Code |
| メンバー | スマホアプリ (React Native / Expo) |

**カレンダー連携:**
- Googleカレンダー: 試合確定時に自動追加（場所・時間・持ち物）
- TimeTree: 共有カレンダーに反映（家族に予定が見える）
- カレンダー側のリマインドも活用できる（アプリ通知と二重で気づく）

**インストール障壁の対策:**
- メンバーに「新しいツールを覚えさせない」原則との矛盾があるが、
  アプリは1回インストールすれば通知タップ→出欠回答の2アクションで完結。
  調整さん等のWebツールより体験は良い。
- LINE公式アカウントのリッチメニューからアプリへの導線を用意することで、
  LINEの延長として使わせる。

## 1. アウトサイドイン: ユーザーストーリーから始める

### 1.1 アクターの定義

| アクター | 誰か | 特徴 | システムとの接点 |
|---------|------|------|----------------|
| **代表** | チームの管理者（1名） | ITリテラシー高い。全業務を一人で回す。Claude Code Max契約あり | Web画面 + Claude Code + LINE |
| **メンバー(プロ)** | 毎週参加の正規メンバー | ITリテラシーはバラバラ。新しいツールは覚えない | LINEのみ |
| **メンバー(ライト)** | 参加頻度低い在籍メンバー | プロと同じ | LINEのみ |
| **助っ人** | 外部の個人（チーム非所属） | チームのシステムにはアクセスしない | LINEのみ（代表経由） |
| **対戦相手チーム** | 他チームの代表 | 自チームと同じ課題を抱えている | メール / LINE / 草野球三番地 |
| **審判** | 派遣審判（S級 / 湘南審判協会） | LINE or 協会HP | LINE / 電話 |
| **自治体予約システム** | 6自治体の公共施設予約サイト | 各自治体でUIもルールも異なる | Webスクレイピング |

### 1.2 コアユーザーストーリー（代表の1週間）

Phase 1で解決すべきストーリーを時系列で並べる。

```
月曜日（4週間前）
  代表: 「来月○日の試合、出欠入れて」→ LINE通知を送信
  メンバー: LINEで「○」「×」「△」をタップして回答

火曜〜木曜（自動）
  システム: 24h未回答者にリマインド送信
  システム: 48h未回答者に個別メンション付きリマインド
  システム: 72h経過で未回答者を「不参加」扱い

金曜日（3.5週間前）
  代表: 「今の状況どう？」→ Claude Codeに聞く
  Claude Code: 「7人確定、2人未回答（過去実績から1人は来る見込み）、
                合計8人予測。あと1人必要」
  代表: 「助っ人探して」
  Claude Code: 助っ人候補DBから「前回来てくれたAさん（○日空き傾向）」を推薦
  代表: 「Aさんに打診して」→ LINEで打診メッセージ送信

同時期
  代表: 「対戦相手探して」
  Claude Code: 相手チームDBから候補を提示 + 打診文面を生成
  代表: 確認して送信

1週間前
  代表: 審判を手配（LINE or 協会HP）
  システム: グラウンド確保状況を確認・通知

前日
  システム: 精算予定をLINE通知
    「明日の費用: グラウンド代¥5,760 + 審判代¥5,000
     相手チームと折半 → ¥5,380
     参加9名 → 一人あたり¥600」

当日
  代表: 参加実績を記録（誰が来た / ドタキャンした）
  システム: PayPay精算リストを生成
    「Aさん: ¥600, Bさん: ¥600, ... （助っ人Xさん: 無料）」

翌日
  代表: 「精算送って」→ PayPayリクエスト or LINE通知
  システム: 立替者への送金情報を整理
```

### 1.3 メンバー側のユーザーストーリー（最小限）

メンバーがやることは**LINEで3タップ以内**で完結する。

```
1. LINEに通知が来る: 「○月○日の試合、出欠を教えてください」
2. 「参加」「不参加」「未定」のボタンをタップ
3. 完了（それ以外は何もしない）
```

メンバーはWebアプリにログインしない。アカウント登録もしない。
LINEのリッチメニューやFlexメッセージで完結する。

### 2.2 システム外のもの（境界の外）

| 対象 | なぜ外か | 連携方法 |
|------|---------|---------|
| 草野球三番地 | プラットフォーム。置き換えない | 手動（募集URLをコピペ） |
| PayPay | 決済API非公開（個人利用） | 精算額を計算してLINE通知。実際の送金は手動 |
| 対戦相手チーム | 相手はシステムを使わない | メール/LINE文面を生成して代表が送信 |
| 審判 | 少人数の個人 | LINE/電話で代表が手配 |
| スポーツ保険 | 年1回の手続き | スコープ外 |

### 2.3 システム内のもの（境界の内）

| 機能 | トリガー | AI要否 |
|------|---------|--------|
| 試合(Game)作成 | 代表がWeb or Claude Codeで作成 | 不要 |
| 出欠通知送信 | Game作成時に自動 | 不要 |
| 出欠回答受付 | メンバーがLINEでタップ | 不要 |
| 段階的リマインド | Cron（24h/48h/72h） | 不要 |
| 締切自動進行 | Cron（締切到来時） | 不要 |
| 出席予測・人数判定 | 代表がClaude Codeで問い合わせ | **必要** |
| 助っ人候補推薦 | 代表がClaude Codeで問い合わせ | **必要** |
| 助っ人打診 | 代表がClaude Codeから指示 | 一部必要（文面生成） |
| 対戦相手打診文面生成 | 代表がClaude Codeから指示 | **必要** |
| 参加実績記録 | 代表がWeb or Claude Codeで入力 | 不要 |
| 精算計算 | 参加実績確定後に自動 | 不要 |
| 精算通知 | 代表が確認後に送信指示 | 不要 |
| グラウンド空き監視 | Cron（定期スクレイピング） | 不要 |
| グラウンド空き通知 | 空き検出時に自動 | 不要 |
| 区分変更提案 | 代表がClaude Codeで問い合わせ | **必要** |
| 週次レポート | 代表がClaude Codeで生成 | **必要** |

### 3.4 試合ライフサイクル（状態遷移）

インタビュー結果に基づき、現行SPEC.mdの状態遷移を修正。

```
DRAFT（下書き）
  │  代表が試合を登録
  ▼
COLLECTING（出欠収集中）
  │  LINE通知送信 → メンバーが回答
  │  自動リマインド（24h/48h/72h）
  │  締切到来 → 未回答を「不参加」扱い
  ▼
ASSESSING（人数判定）
  │  参加人数が足りるか？
  │  ├─ 足りる → ARRANGING へ
  │  └─ 足りない → 助っ人打診 → 助っ人確定後 ARRANGING へ
  ▼
ARRANGING（手配中）
  │  対戦相手確定 / グラウンド確定 / 審判手配
  │  全て揃ったら →
  ▼
CONFIRMED（確定）
  │  前日: 精算予定通知
  │  当日: 試合実施
  ▼
COMPLETED（完了）
  │  参加実績記録 → 精算計算 → 精算通知
  ▼
SETTLED（精算済み）

分岐:
  COLLECTING → CANCELLED（人数不足で中止）
  ARRANGING  → CANCELLED（相手見つからず / グラウンド取れず）
  CONFIRMED  → CANCELLED（雨天中止等）
```

### 4.2 Vercel vs exe.dev の判断

| 項目 | Vercel | exe.dev |
|------|--------|---------|
| Next.js ホスティング | 得意 | 可能 |
| Cron Worker | 制限あり（Hobby: 1日1回） | 自由 |
| LINE Webhook 常時受信 | Serverless（コールドスタート） | 常時起動可 |
| 費用 | 無料枠あり | 契約済み |

**判断:** exe.devをメインのホスティングとし、Vercelは不要（exe.devでNext.jsもCronも動かせる）。
ただし、Vercelの方が便利であればVercel(フロント) + exe.dev(Worker)の分離も可。

### 4.3 AI実行の制約と設計

```
【絶対ルール】
  ・AIの実行は代表がClaude Codeを起動したときのみ
  ・サードパーティプログラムからClaude Codeを自動呼び出しすることは禁止（TOS違反）
  ・Open Claw等の非公式クライアントは使用しない

【設計方針】
  ・ルールベースで自動化できるものはexe.devのCronで実行（AIなし）
  ・判断が必要なものは代表がClaude Codeで手動実行
  ・Claude CodeはSupabaseのデータを読み、判断結果をSupabaseに書き、LINE APIで通知

【AI不要で自動化するもの】
  ・出欠リマインド送信（ルール: 24h/48h/72h後に未回答者へ送信）
  ・出欠締切処理（ルール: 締切到来時に未回答を不参加扱い）
  ・精算額計算（ルール: 費用÷参加メンバー数。助っ人は除外）
  ・グラウンド空き通知（ルール: 空きスロット検出時にLINE通知）

【AI必要で代表が手動実行するもの】
  ・出席予測（過去データからの統計推論 + 自然言語レポート）
  ・助っ人推薦（候補のスコアリング + 推薦理由の説明）
  ・打診文面生成（相手チームへの自然な文面）
  ・メンバー区分変更提案（参加率分析 + 変更理由の言語化）
```

### 4.4 セキュリティ・プライバシー設計

```
【最小限の個人情報】
  ・メンバー: 名前（ニックネーム可）+ LINE User ID のみ
  ・住所・電話番号・家族情報は保持しない（過去に問題になった）
  ・助っ人: 名前 + LINE連絡先 のみ

【認証】
  ・代表のみSupabase Auth でログイン（Web Dashboard）
  ・メンバーはLINEの認証で識別（LINE User ID）
  ・メンバーがWebアプリにログインすることはない

【データアクセス】
  ・Supabase RLS: 代表のチームのデータのみ参照可能
  ・LINE Webhook: 署名検証で不正リクエスト排除
  ・Claude Code: 代表のマシンからのみSupabaseにアクセス
```

## 5. 技術スタック（確定）

| レイヤー | 技術 | 理由 |
|---------|------|------|
| フロントエンド | Expo (React Native) | iOS / Android / Web を1コードベースで |
| ルーティング | Expo Router | ファイルベース。Next.js App Routerと同じ感覚 |
| UI | NativeWind (Tailwind for RN) | Tailwind記法をReact Nativeで使える |
| 言語 | TypeScript (strict) | 既存実装を活かす |
| DB | Supabase PostgreSQL | SQLの強み（精算集計・統計）。無料枠十分 |
| 認証 | Supabase Auth | Google / Apple / LINE（カスタムOIDC） |
| クエリ | @supabase/supabase-js | Expoではssr不要。通常クライアント |
| プッシュ通知 | Expo Notifications | 無料・無制限。iOS/Android両対応 |
| サーバーサイドロジック | Supabase Edge Functions | Webhook受信、Cronジョブ（リマインド、締切処理等） |
| カレンダー連携 | Google Calendar API / TimeTree API | 試合確定時に自動追加 |
| ホスティング(Web) | exe.dev or Vercel | Expo for Webの配信 |
| グラウンド監視 | ground-reservation (Python) | 既存実装。exe.devに統合 |
| AI判断 | Claude Code Max | TOS準拠。代表が手動実行 |
| パッケージマネージャ | Bun | 既存実装 |
| Linter | Biome | 既存実装 |
| CI/CD | GitHub Actions + EAS Build | アプリビルドはExpo Application Services |
| テスト | Bun test (BDD) | 既存実装 |

### モノレポ構成（新）

```
packages/
  core/       — ドメイン型定義、ステートマシン、ビジネスロジック（既存、共有）
  app/        — Expo アプリ（iOS / Android / Web）← 新規
  web/        — Next.js（代表用ダッシュボード）← 既存、段階的にExpo Webに統合
supabase/
  migrations/ — DBスキーマ
  functions/  — Edge Functions（Webhook、Cron）← 新規
  seed.sql    — テストデータ
```

`packages/core` はフロントエンド（Expo）とサーバーサイド（Edge Functions）の両方から参照される。
