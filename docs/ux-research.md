# mound UX Research & Recommendation

世界最高のプロダクトUXから学び、mound の体験設計に活かす。

---

## 1. Conversational AI Products

### ChatGPT

**ONE THING:** 空のテキストボックスが全ての入り口になる。何でも書ける自由度が、ユーザーの心理的障壁を極限まで下げている。

- **入力:** 大きなテキストエリアがページ中央に鎮座。プレースホルダーに「Message ChatGPT...」とだけ表示。ファイル添付・画像・音声入力もテキストボックスに統合
- **サジェスト:** 初回は4つのサンプルプロンプトをカード形式で提示。会話が進むとコンテキストに応じたフォローアップ質問を自動提示
- **結果表示:** ストリーミングで文字が流れる。Markdown レンダリング、コードブロック、テーブル、画像生成結果を会話内にインライン表示。Canvas 機能で文書やコードを右ペインに分離表示
- **摩擦の削減:** ログイン不要で使い始められる。会話履歴がサイドバーに自動保存。GPT Store でカスタムエージェントを共有可能

**mound への示唆:** 代表が「4/11 練習試合 9時から」と自然言語で入力するだけで全てが動く、という mound のビジョンに直結。テキストボックスを入り口にし、構造化データへの変換はAIが裏でやる。

### Claude

**ONE THING:** Artifacts（成果物）の概念。会話と成果物を明確に分離し、右ペインに独立したドキュメント・コード・図を生成する。会話は「プロセス」、Artifacts は「結果」。

- **入力:** シンプルなテキストエリア。プロジェクト機能でコンテキストを事前に設定可能
- **結果表示:** 会話の流れを壊さず、成果物を別ペインに表示。成果物は独立してコピー・編集・ダウンロード可能
- **摩擦の削減:** Projects 機能で繰り返しのコンテキスト設定を不要に。Memory 機能でユーザーの好みを学習

**mound への示唆:** 会話（指示・相談）と結果（試合カード・出欠表・精算表）を分離する設計。代表がAIと会話しながら、右側に試合の状況がリアルタイムで更新されるUI。

### Perplexity

**ONE THING:** 答えにソースが付く。情報の信頼性を視覚的に担保する「引用付き回答」が、検索エンジンとチャットの間の新しいカテゴリを作った。

- **入力:** 検索バー + フォーカスモード（Academic, Writing, etc.）でコンテキストを絞れる
- **結果表示:** 回答本文に [1][2][3] のインライン引用。関連質問を自動提示。Pro Search では思考プロセスを可視化
- **摩擦の削減:** 質問するだけで、複数のソースを横断して統合された回答が得られる

**mound への示唆:** AIの提案に「根拠」を付ける。「9人集まる見込みです」ではなく「過去6回の参加率から山田(90%)、佐藤(85%)、田中(70%)が参加見込み → 9人予測」と根拠付きで提示する。

### Google Gemini

**ONE THING:** マルチモーダル統合。テキスト・画像・動画・コードを区別なく扱えるインターフェース。Google Workspace との深い統合でメール・カレンダー・ドキュメントを横断的に参照。

- **入力:** テキスト + 画像 + ファイルアップロードを統一的に受付。Extensions で Gmail, Maps, YouTube 等を直接呼び出し
- **結果表示:** テキスト + 画像生成 + コード実行結果を統合表示。Google Workspace 連携で直接ドキュメントを操作
- **摩擦の削減:** 既存の Google エコシステムと深く統合。新しいツールを覚える必要がない

**mound への示唆:** LINE・Google Calendar・PayPay 等、メンバーが既に使っているツールとの深い統合。新しいアプリを覚えさせるのではなく、既存の生活動線の中に mound を埋め込む。

## 2. Team/Group Management Tools

### Slack

**ONE THING:** チャンネルベースの文脈分離。話題ごとにチャンネルを分け、情報の洪水を防ぐ。スレッドで深い議論を本流から分離。

- **ボットとの共存:** /slash コマンドでボットを呼び出し。ボットのメッセージは人間と同じ流れに表示されるが、アイコンとラベルで区別。インタラクティブなボタン・セレクタをメッセージ内に埋め込める
- **摩擦の削減:** @メンションで必要な人だけに通知。ステータス設定でオンライン状態を共有。絵文字リアクションで軽量なフィードバック
- **情報アーキテクチャ:** サイドバーにチャンネル一覧。スター・ブックマークで重要情報をピン。検索が強力

**mound への示唆:** 試合ごとに「チャンネル」を自動生成し、出欠・連絡・精算を1箇所にまとめる。絵文字リアクション的な軽量レスポンス（参加/不参加をボタン1つ）。

### Discord

**ONE THING:** ロールベースの権限とコミュニティ形成。サーバー内で細かいロール設定ができ、大規模コミュニティでも秩序が保たれる。

- **ボットとの共存:** Bot は専用チャンネルを持てる。スラッシュコマンドでインタラクティブに操作。ボットがボタン・セレクトメニュー・モーダルフォームを送信できる
- **独自の工夫:** ステージチャンネル（音声）、フォーラムチャンネル（構造化ディスカッション）、イベント機能でスケジュール管理
- **摩擦の削減:** 招待リンク1つでサーバーに参加。モバイルアプリの通知カスタマイズが細かい

**mound への示唆:** Discord のフォーラムチャンネルのように、各試合が独立したスレッドとして管理される構造。「試合フォーラム」という考え方。

### LINE（グループ）

**ONE THING:** 日本で最も低い参加障壁。アプリをインストールしていない日本人がほぼいない。グループ招待がQRコードで完結。

- **グループでのボット:** LINE Official Account がグループに参加し、メッセージを受信・送信。リッチメニューで常設のUIを提供。Flex Message で複雑なカード型UIを送信可能
- **制約:** グループ内の既読が誰か分からない。メッセージが流れやすい（ピン留め機能が弱い）。Push API の無料枠が200通/月と少ない
- **摩擦の削減:** QRコードで参加。アカウント作成不要。生活インフラとして既に使われている

**mound への示唆:** mound の最重要チャネル。メンバーの出欠回答は LINE 上で完結させるべき。ただし情報の永続性（ピン留め・検索）はWebアプリで補完する。LINE はトリガー、Web は情報ストア。

### WhatsApp

**ONE THING:** エンドツーエンド暗号化による信頼感と、シンプルさの徹底。Channels 機能でブロードキャスト型の情報発信が可能。

- **グループ管理:** 最大1024人のグループ。Communities 機能でグループのグループを作成可能。アナウンス専用グループで情報の流れを制御
- **摩擦の削減:** 電話番号だけで始められる。ステータス機能で軽量な近況共有

**mound への示唆:** WhatsApp Communities のように、チーム > 試合 > 出欠 の階層構造でグループを自動管理する発想。

## 3. Calendar/Scheduling Tools

### Google Calendar

**ONE THING:** 時間軸上の可視化。月・週・日ビューの切り替えで、スケジュールの全体像と詳細を自在に行き来できる。

- **スケジュール表示:** カラーコーディングで種類を区別。他人のカレンダーを重ねて表示し、空き時間を瞬時に把握
- **グループスケジューリング:** 「おすすめの時間」機能でメンバー全員の空き時間を自動提案。会議室の空き状況も統合表示
- **摩擦の削減:** Gmail のメール内容から自動でイベント作成を提案。場所を入力すると移動時間を自動計算

**mound への示唆:** 試合スケジュールをカレンダービューで表示し、出欠状況をカラーで可視化。メンバーの「参加可能日」を重ねて表示し、最適な日程をAIが提案。

### Calendly

**ONE THING:** 「空いている時間を見せる」という発想の転換。自分のカレンダーから空き時間だけを外部に公開し、相手に選ばせる。

- **フロー:** ホストが空き時間ルールを設定 → 共有リンクを送る → ゲストが日時を選ぶ → 両者のカレンダーに自動追加
- **グループ:** Round Robin（順番に割り当て）、Collective（全員が空いている時間のみ表示）
- **摩擦の削減:** ゲストはアカウント不要。リンク1つで予約完了。リマインドメールが自動送信

**mound への示唆:** 対戦相手との日程調整を Calendly 型で実現。チームの空き時間を公開し、相手チームに選ばせる。「mound.app/match/ABC123」のようなリンクで日程調整が完結。

### cal.com

**ONE THING:** オープンソースかつ高度にカスタマイズ可能。ワークフロー機能で予約前後のアクションを自動化（メール送信、Webhook、SMS等）。

- **ワークフロー:** 「予約確定時にSlackに通知」「前日にリマインドメール」「キャンセル時にウェイティングリストの次の人に通知」を GUI で設定
- **チーム機能:** チームページで全メンバーの空き時間を統合表示
- **摩擦の削減:** セルフホスト可能で、カスタムドメインで使える

**mound への示唆:** cal.com のワークフロー設計が、mound の「出欠確認 → リマインド → 助っ人打診 → 確定」のフローに直結。イベントドリブンな自動化をGUIで設定可能にする。

### TimeTree

**ONE THING:** 家族・グループで「同じカレンダーを見る」体験。個人のカレンダーではなく、共有カレンダーが主役。

- **共有:** カレンダー作成 → リンクで招待 → 全員が同じビューを見る。イベントにコメント・画像を付けられる
- **独自の工夫:** イベントにチャットが紐づく。メモ機能でイベント詳細を整理。Keep（お知らせ板）機能で永続的な情報を固定表示
- **摩擦の削減:** 家族間のスケジュール共有が主用途。直感的なUI。招待リンクで参加

**mound への示唆:** チーム全体のカレンダーを共有し、試合・練習・イベントを一覧表示。各試合にチャットスレッドが紐づき、出欠・連絡・精算が1箇所にまとまる。

## 4. Sports Team Apps

### TeamSnap

**ONE THING:** スポーツチーム運営に特化した「全部入り」。出欠管理・スケジュール・メッセージ・写真共有・精算を1つのアプリに統合。

- **出欠管理:** 試合・練習ごとに出欠を記録。保護者（子供のスポーツ向け）が代理回答可能。出欠一覧がダッシュボードで一目瞭然
- **スケジュール:** カレンダービュー + リスト形式。フィルタリング（種類別・チーム別）。iCal連携で外部カレンダーに同期
- **課題:** UIが複雑。機能が多すぎて初見では迷う。北米市場向けで、日本の草野球文化（助っ人、精算、LINE依存）に合わない。**「何でもできる」は「何をすればいいか分からない」になりやすい**

**mound への示唆:** TeamSnap の「全部入り」を反面教師にする。mound は「必要十分な情報だけ」を表示し、今やるべきことだけにフォーカスさせる。

### GameChanger

**ONE THING:** リアルタイムスコアリング。試合中にスコアを入力すると、遠隔地の家族やファンがライブでスコアを見られる。スタッツの自動集計。

- **スコアリングUI:** 野球に特化したスコアリングインターフェース。投球数・打球方向・走塁をタップで記録。リアルタイムでボックススコアを更新
- **統計:** 打率・防御率・OPS等を自動計算。シーズン通算成績をグラフ表示。チームランキング
- **ソーシャル:** ライブストリーミング機能。試合のハイライト動画を自動生成。SNS共有機能
- **課題:** スコアリングの入力者の負担が大きい。草野球では審判もスコアラーもいないことが多い

**mound への示唆:** 試合結果・個人成績の記録機能は Phase 2 以降で。ただし、メンバーが自分のスタッツを見られることはモチベーション維持に有効。最初は最小限（勝敗・スコア・MVP投票程度）から。

### BANDS

**ONE THING:** コミュニティプラットフォーム。掲示板・出欠・カレンダー・写真・投票を1つのバンド（グループ）にまとめる。韓国発で、日本のスポーツチームにも利用者がいる。

- **出欠管理:** イベントに出欠投票を紐づけ。参加者一覧を視覚的に表示
- **コミュニケーション:** 掲示板形式でトピック別に投稿。チャット機能もあり。投票機能で意思決定
- **課題:** 汎用的すぎてスポーツ特化のUXではない。日本での認知度が低い

**mound への示唆:** BANDS のように掲示板（永続情報）+ チャット（フロー情報）+ イベント（スケジュール）を統合する発想は正しいが、スポーツに特化した情報アーキテクチャが必要。

### 草野球三番地（草野球.net 系）

**ONE THING:** 対戦相手マッチング。地域・レベル・日程で相手チームを検索し、対戦を申し込める。日本の草野球文化に特化した唯一のプラットフォーム。

- **マッチング:** 地域・曜日・レベルで検索。掲示板形式で対戦募集を投稿。マッチング成立後の連絡はメール
- **グラウンド情報:** 球場データベース。利用料金・アクセス・設備の情報
- **課題:** UIが古い（2000年代のWeb）。マッチング後のフローが手動（メールでやり取り）。モバイル最適化されていない。チーム運営機能は皆無。**マッチングの「後」が完全に抜けている**

**mound への示唆:** 草野球三番地の「マッチングの穴」を埋めることが mound の Phase 2 の核心。マッチングから確定までをシームレスに自動化する。ただし Phase 1 では自チーム運営の完成度を上げることが先。

## 5. Magic/Zero-config Products

### Notion AI

**ONE THING:** コンテキスト内AI。ドキュメントの内容を理解した上で、要約・翻訳・続きの執筆・表の生成をインラインで行う。AI がエディタの中に溶け込んでいる。

- **呼び出し方:** スペースキーで AI を呼び出し。テキストを選択して AI メニュー。/ コマンドで AI アクション。ページ横の AI ボタン
- **結果表示:** 生成結果がドキュメント内にインラインで挿入される。既存のブロック構造に自然に統合
- **摩擦の削減:** AIの出力が Notion のブロック形式で出るので、そのまま編集・移動・削除できる

**mound への示唆:** AI を別画面に分離するのではなく、管理画面のコンテキスト内に AI を埋め込む。試合詳細画面で「この試合の状況を分析して」と聞くと、その場で分析結果が表示される。

### Linear

**ONE THING:** 速度。全てのインタラクションが瞬時に反応する。オプティミスティックUI更新、キーボードショートカット、バッチ操作。生産性ツールの速度の新基準を作った。

- **速度の秘密:** ローカルキャッシュ + オプティミスティック更新。サーバーとの同期を待たずにUIを即座に更新。オフライン対応
- **ナビゲーション:** Cmd+K でグローバル検索・コマンドパレット。キーボードだけで全操作可能。ビュー切り替え（Board / List / Timeline）がシームレス
- **オンボーディング:** プロジェクトを作るだけで、ワークフロー（Backlog → Todo → In Progress → Done）が自動設定。Git 連携でPRとイシューが自動リンク
- **デザイン:** 抑制された色使い。ステータスが色アイコンで瞬時に判別。密度の高い情報表示だが圧迫感がない

**mound への示唆:** Linear の速度哲学を mound に。出欠回答はタップした瞬間に反映（オプティミスティック更新）。代表の管理画面はキーボードショートカット対応。**ローディングスピナーは体験を壊す。**

### Superhuman

**ONE THING:** スプリットインボックス。メールを自動でカテゴリ分け（Important / Other / Feed）し、処理済みにすることでインボックスゼロを達成させる。

- **速度:** 100ms 以内のレスポンス。先読みで次のメールを事前ロード。キーボードショートカットで全操作
- **AI 機能:** メールの要約。返信文面の自動生成。「3行で」「フォーマルに」等のトーン指定
- **オンボーディング:** 1対1のオンボーディングコール。使い方を人間が教える（月30ドルの価値を実感させる）
- **ゲーミフィケーション:** インボックスゼロ達成時のお祝い画面。処理速度のスタッツ表示

**mound への示唆:** 代表のダッシュボードを「インボックス型」にする。未処理タスク（未確認の出欠、未手配の助っ人、未確保のグラウンド）をリスト化し、1つずつ処理していくフロー。全てクリアすると「今週の準備完了!」と表示。

### Arc Browser

**ONE THING:** サイドバー型ナビゲーション + Space の概念。ブラウザのタブをサイドバーに縦に並べ、Space（仕事 / 個人 / プロジェクト）でコンテキスト切り替え。

- **スペース:** 用途別にタブセットを切り替え。各スペースで異なるプロフィール（アカウント）
- **Boost:** Webサイトの外観をカスタマイズ。フォント・色をユーザーが変更可能
- **コマンドバー:** Cmd+T でURLも検索もコマンドも全て1つのバーから実行
- **自動アーカイブ:** 一定時間使わないタブは自動でアーカイブ。タブの散らかりを防ぐ

**mound への示唆:** 代表が複数チームを管理する場合の「スペース切り替え」。試合が終わったら自動的にアーカイブされ、アクティブな試合だけが表示される。

### Raycast

**ONE THING:** ランチャーから全てを操作。Cmd+Space で起動し、アプリ起動・ファイル検索・計算・翻訳・スニペット・ウィンドウ管理を1つのインターフェースで。

- **拡張性:** Store でサードパーティ拡張を追加。GitHub, Jira, Linear, Notion 等と統合
- **AI:** チャット機能が内蔵。システム全体のコンテキスト（クリップボード、選択テキスト等）を AI に渡せる
- **スニペット:** 定型文を短縮キーで展開。変数（日付、クリップボード等）を埋め込み可能
- **摩擦の削減:** アプリ間の行き来が不要。1つの入力欄から全てにアクセス

**mound への示唆:** 代表の管理画面に Raycast 型コマンドパレット。Cmd+K で「試合作成」「出欠状況確認」「助っ人打診」「精算」をどこからでも呼び出せる。

## 6. Social Platforms with Discovery

### TikTok（For You）

**ONE THING:** ゼロフォローでもコンテンツが届く。アカウント作成した瞬間から、アルゴリズムが好みを推測してコンテンツを提供。フォローグラフに依存しない発見エンジン。

- **発見の仕組み:** 全画面動画 → スワイプで次 → 視聴時間・いいね・コメントで好みを学習 → フィードが最適化
- **コンテンツクリエイターの平等性:** フォロワー0でもバイラルする可能性がある。コンテンツの質だけで勝負できる
- **摩擦の削減:** 開くだけでコンテンツが始まる。意思決定は「見続けるか、スワイプするか」だけ

**mound への示唆:** Phase 3 のチーム間マッチングで、チームの「For You」フィードを実装。レベル・地域・日程が合うチームの情報が自動的にタイムラインに流れてくる。チームは「投稿」するだけで良い。

### Instagram Explore

### 7.4 ナビゲーション: コンテキスト駆動

**代表向け（Web Dashboard）:**
- **サイドバー + コマンドパレット（Linear 型）。** 左サイドバーにセクション（要対応 / カレンダー / チーム / 設定）。Cmd+K でどこからでもアクション実行
- **ドリルダウン:** リスト → 詳細 → アクション の3レベル。パンくずリストで現在位置を表示
- **ゼロクリック判断:** ダッシュボードに表示される情報だけで、80%の判断が完結する密度

**メンバー向け（スマホアプリ）:**
- **タブバー（4つだけ）。** ホーム / カレンダー / プロフィール / 設定。メンバーに複雑なナビゲーションは不要
- **ホーム = 次の試合。** アプリを開いた瞬間に「次に何をすべきか」が分かる
- **通知からのディープリンク:** プッシュ通知をタップ → 出欠回答画面に直接遷移 → 1タップで回答完了

### 7.5 オンボーディング: ゼロコンフィグ + プログレッシブ

**代表のオンボーディング（3ステップ、3分で完了）:**

```
Step 1: チーム名を入力         「チーム名は？」
Step 2: メンバーを招待          LINE/メールで招待リンクを送信
Step 3: 最初の試合を作成        「次の試合はいつ？」→ 自然言語で入力

→ 以上。設定は後からいつでも変更可能。
```

**やらないこと:**
- ❌ 長いチュートリアル
- ❌ 全機能の説明
- ❌ 設定画面の巡回
- ❌ プロフィールの詳細入力

**プログレッシブ開示（使い始めてから段階的に）:**

```
1回目の試合: 出欠管理だけ体験
2回目の試合: 「リマインドを自動送信しますか？」と提案
3回目の試合: 「助っ人機能を使ってみませんか？」と提案
5回目の試合: 「精算を自動化できます」と提案
```

機能は必要になったタイミングで、コンテキストに合わせて提案する。

**メンバーのオンボーディング（ゼロステップ）:**

```
1. LINE/メールで招待リンクが届く
2. タップしてログイン（Google/LINE）
3. チームに自動参加
4. 次の試合の出欠回答画面が即表示

→ アカウント作成という概念すらない
```

### 7.6 Multi-user Interaction: AI と人間が同じ場にいる

**設計原則:**
1. AI のアクションは人間と区別可能だが、同じ流れの中にある
2. AI は「提案」し、人間が「承認」する。この境界を視覚的に明示
3. AI の思考プロセスは折りたたみで詳細表示可能（Perplexity 型）

## 11. まとめ: mound の UX ビジョン

> **「伝えるだけで動く。覚えることはない。必要十分な情報だけ。」**

mound は 2 つの異なる UX を 1 つのプロダクトに共存させる:

1. **代表向け: AI パートナーと対話する管理ツール**
   - Linear の速度 + Superhuman のインボックス + ChatGPT の自然言語入力
   - 「状況を聞く → 提案を受ける → 承認する」の3ステップで全てが回る

2. **メンバー向け: 通知が来たらタップするだけのアプリ**
   - LINE の手軽さ + Calendly のゲスト体験
   - アカウント作成不要、覚えることゼロ、出欠回答は1タップ

この二面性こそが mound の本質的な差別化要因。既存のスポーツチームアプリは全員に同じ UI を提供するが、mound は役割に応じて体験を最適化する。

**最も重要な3つの UX 原則:**

1. **Speed is a feature** — 全てのインタラクションが 100ms 以内に応答する
2. **AI proposes, human disposes** — AI は提案し、人が承認する。その境界を常に明示する
3. **Meet users where they are** — LINE ユーザーには LINE で、アプリユーザーにはアプリで。ユーザーに合わせる
